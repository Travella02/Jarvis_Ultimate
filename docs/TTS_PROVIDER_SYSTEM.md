# TTS Provider System

Version: 0.0.7c

Jarvis voice output is built as a swappable provider layer. The brain should not
know whether speech is generated by Kokoro, ElevenLabs, Windows TTS, XTTS, or a
mock provider. The TTS manager owns provider selection, fallback, output files,
voice selection, and optional playback.

## Current provider strategy

- `kokoro` is now the default local TTS provider. It already worked on Tanner's
  laptop, is much simpler than XTTS to keep stable, and is the cleaner direction
  for a future SaaS-ready Jarvis.
- `mock` is always the fallback so tests and fresh installs still work even when
  optional TTS packages are not installed.
- `xtts` remains in the codebase only as an experimental personal/local provider.
  It is not part of the default chain after 0.0.7c.
- `elevenlabs` can be added later as a premium/cloud provider without changing
  the brain because the provider layer is already swappable.

## SaaS/commercial licensing note

XTTS v2's public model license is non-commercial and covers model outputs. Keep
XTTS out of the default path unless a separate commercial license is secured. If
Jarvis becomes SaaS/commercial, prefer Kokoro locally or add a licensed provider
such as ElevenLabs.

## Configuration

The main settings live in `.env` or `config/providers.yaml`:

```env
JARVIS_TTS_ENABLED=true
JARVIS_TTS_PROVIDER=kokoro
JARVIS_TTS_FALLBACK_PROVIDERS=mock
JARVIS_TTS_OUTPUT_DIR=data/tts
JARVIS_TTS_VOICE_NAME=jarvis
JARVIS_TTS_LANGUAGE=en
JARVIS_TTS_DEVICE=auto
JARVIS_TTS_USE_GPU=false
JARVIS_TTS_PLAYBACK=false
JARVIS_TTS_AUTO_SPEAK=false
JARVIS_TTS_KOKORO_VOICE=af_heart
JARVIS_TTS_KOKORO_LANG_CODE=a
```

`JARVIS_TTS_VOICE_NAME=jarvis` is the assistant identity. The actual Kokoro
speaker voice is `JARVIS_TTS_KOKORO_VOICE`.

## Install on another computer

From the Jarvis root with the venv active:

```powershell
python -m pip install -r requirements-tts.txt
```

That file is now Kokoro-focused. It avoids the XTTS dependency chain that caused
CUDA, Transformers, NumPy, Torch checkpoint, TorchCodec, and FFmpeg issues.

If you intentionally want to revisit XTTS later, use the separate experimental
file instead:

```powershell
python -m pip install -r requirements-tts-xtts-experimental.txt
```

## CLI commands

```text
tts status
tts providers
tts test
tts test play
tts say Hello sir. Jarvis voice output is connected.
tts play last
tts playback on
tts playback off
voice on
voice off
```

`voice on` enables automatic speech and playback for the current CLI runtime
session. It does not permanently edit `.env`.

## Kokoro voices

Useful commands:

```text
tts voice list
tts voice current
tts voice use af_bella
tts voice test af_bella play
tts say as af_bella Hello sir.
```

Jarvis lists common Kokoro voice IDs, but you may also try provider-specific
voice IDs if your installed Kokoro package supports them.

## XTTS legacy/experimental commands

These commands still exist for debugging, but they are not the recommended path:

```text
tts reference
tts reference import C:\path\to\reference.wav
tts xtts test play
tts debug last
```

Use XTTS only for personal/local experiments with voices you own or have clear
permission to use. Do not build the SaaS/commercial Jarvis path on XTTS unless a
commercial license is obtained.
